OKF Project Blueprint

A no-code project definition for what we are making, why it should exist, what the machine needs, what already exists, and what belongs on GitHub.

Date: 2026-07-04 Phase: product definition Coding: not yet

Decision

Make an open-source OKF context packer for agents.

Start with local files: create, validate, index, retrieve, and pack context from Open Knowledge Format bundles. The viewer is a human inspector, not the AI runtime.

One Sentence

Knowpack helps agents turn OKF bundles into compact, cited, task-ready context.

This is not a Google Cloud clone, not a database catalog, and not a hosted platform. It is a repo-native tool for knowledge files that agents can safely read.

The Problem

Mess

Knowledge is scattered

Important context lives in docs, schemas, READMEs, tickets, dashboards, Slack notes, source comments, and human memory.

Risk

Agents guess

Raw schemas and docs rarely say what matters: source of truth, caveats, deprecated paths, units, definitions, and proof.

Gap

Existing tools are heavy

Catalog platforms are useful, but many projects need a small file-first format and validator before they need a platform.

The User

User What they need Why OKF helps
Solo developer Make an agent understand a project without huge prompts. Store durable context as small linked Markdown files.
Data engineer Explain tables, metrics, sources, joins, and caveats. Wrap schemas with business meaning and citations.
AI agent builder Give tools and models trusted local context. Expose validated bundles through file reads, search, graph, or MCP.
Open-source maintainer Package project knowledge for contributors and agents. Keep the knowledge beside the code and review it in git.

The Machine

The full machine has six parts. The agent path matters most; the viewer is only a way for humans to inspect what the agent will read.

1. Format Markdown files with YAML frontmatter, links, citations, and reserved files like index.md.
2. Validator Checks bundle structure, frontmatter, required fields, broken links, citations, and warnings.
3. Indexer Builds index.md, graph JSON, backlinks, search metadata, and summaries.
4. Retriever Finds relevant concepts by title, type, tags, headings, body text, links, and backlinks.
5. Context Packer Returns the smallest useful context pack with summaries, caveats, sources, and citations.
6. Viewer Static HTML inspector for humans: docs, search, graph, links, citations, and errors.

What We Are Making

Version 0.1

  • A CLI that validates an OKF bundle.
  • A command that creates new concept files from templates.
  • An indexer that generates navigation and graph metadata.
  • A search/context command that returns agent-ready context.
  • A static HTML inspector for humans.
  • Fixture bundles that prove what passes and fails.

Not Version 0.1

  • No hosted service.
  • No Google Cloud dependency.
  • No BigQuery integration unless used only as a converter demo.
  • No crawler that reads the whole internet.
  • No new ontology standard.
  • No custom database.

Existing Projects And Our Angle

Thing What it does Our angle
GoogleCloudPlatform OKF Draft spec, samples, reference agent, visualizer. Focus on context packing and retrieval evaluation rather than duplicating generic CLI/lint behavior.
Google Cloud Knowledge Catalog Managed enterprise product for metadata, governance, semantic search, and agent context. Stay local and open-source first. Integrate later only if useful.
llms.txt Markdown file convention for LLM-friendly site summaries. OKF is multi-file, typed, linked, cited, and graphable.
Backstage Developer portal and software catalog. OKF is lighter and agent-readable without running a portal.
OpenMetadata and DataHub Full metadata platforms. Knowpack is for model-ready context packs and retrieval tests, not platform governance.
CKAN and Data Package Open data publishing and dataset packaging. OKF can consume them and add agent-use instructions and caveats.

Risks

Risk

Spec changes

OKF is draft. Our tool should preserve unknown keys and avoid pretending the format is final.

Risk

Too broad

If we try to support every catalog and agent from day one, the project becomes noise.

Risk

Name confusion

OKF also means Open Knowledge Foundation in many contexts. The README must define our meaning immediately.

Proof Before Code

  1. Write the exact validation rules.
  2. Create the billing OKF demo bundle by hand.
  3. Write expected validation output for the demo.
  4. Write the README pitch and comparison table.
  5. Only then implement the CLI against those examples.
If the hand-written demo does not feel useful, code will not save the project. Fix the concept first.
This blueprint is a planning artifact. It intentionally avoids implementation code.
