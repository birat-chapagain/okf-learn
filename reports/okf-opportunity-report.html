Bottom Line

What OKF is

Open Knowledge Format is a draft format from the GoogleCloudPlatform knowledge-catalog repo. It represents knowledge as a directory of UTF-8 Markdown files with YAML frontmatter. A concept can be a table, API endpoint, metric, playbook, reference, business rule, service, policy, or any other unit an agent needs to understand.

Where the opportunity is

Google's repo proves the idea with samples, a reference agent, tests, and a self-contained HTML visualizer. The remaining gap is proving that an OKF bundle retrieves the right concepts, caveats, and citations for a model question.

OKF is not only for databases and not only for Google Cloud. It is plain files. Google Cloud Knowledge Catalog is a managed product; OKF is a portable file convention that can sit in any repo.

What We Have

Strong

Draft spec

OKF v0.1 defines bundle structure, concept documents, frontmatter, cross-links, index files, logs, citations, and conformance rules.

Strong

Concrete examples

The Google repo includes generated bundles for GA4, Stack Overflow, and Bitcoin public datasets, plus visualizer artifacts.

Partial

Producer pattern

The reference agent shows a two-pass path: gather BigQuery metadata first, then use web seeds to enrich concepts with source docs.

Partial

Viewer pattern

The repo has a self-contained HTML visualizer with graph, search, filters, detail panel, rendered markdown, and backlinks.

Strong

Clear license

The GoogleCloudPlatform knowledge-catalog repo is Apache-2.0 licensed, while also saying the repo is not an official Google product.

Local

Our notes

This repo now has OKF study notes, skill notes, source map, and Google OKF repo notes. Those are enough to start a focused implementation.

What We Do Not Have

Missing

Context evaluator

Early OKF CLIs and generators exist now. The stronger missing piece is a small tool that tests query-to-context quality for agents.

Missing

Machine schema

The spec describes YAML frontmatter, but there is no official JSON Schema for validation or tool generation.

Thin

Typed relationships

Markdown links create graph edges, but relationship type is only implied by nearby prose. That is flexible, but weak for automation.

Thin

Canonical types

The spec intentionally avoids a registry of concept types. That is good for adoption, but a validator cannot know if "Metric" or "KPI" is preferred.

Missing

Security model

Rendering untrusted markdown, following links, reading private files, and exposing bundles to agents need clear rules.

Missing

Compatibility suite

No official set of pass/fail fixture bundles exists yet. An open-source tool can provide this and become useful quickly.

What Can Be Made Now

The first useful product is not another catalog and not another generic OKF linter. It is a local context packer that tests whether an OKF bundle can feed the right facts to a model.

1. Sources Markdown, docs, OpenAPI, CKAN, Data Package, schemas, CSV notes.
2. Convert Create concept files with frontmatter, body, links, and citations.
3. Validate Check frontmatter, reserved files, links, citations, and bundle shape.
4. Index Generate index.md, graph JSON, search metadata, and backlinks.
5. Consume Render HTML, serve via MCP, or load into an agent context.

Context packer Given a query, retrieves relevant concepts, caveats, links, resources, and citations into compact model-ready JSON. This is the real agent path: OKF bundle to model context. Low
Retrieval evaluator Runs golden tests: query, required concepts, forbidden concepts, required citations, required caveats. Proves the bundle is useful for agents, not just syntactically valid. Low
OKF new Creates concept files from templates: Metric, API Endpoint, Table, Playbook, Reference, Policy. Reduces messy hand-authored files and encourages consistent structure. Low

MCP server Exposes OKF bundle search/read/graph as MCP resources and tools. Lets Claude, ChatGPT, Cursor, VS Code, and other MCP clients consume OKF context. Medium
Converters Imports OpenAPI, CKAN, Data Package, dbt docs, and simple folder docs into OKF concepts. Bootstraps real projects faster than asking humans to write every concept manually. Medium

Recommended MVP

Ship this first

knowpack context <bundle> "can this customer get a refund?"
knowpack eval <bundle> retrieval-tests.yml
knowpack inspect <bundle>
knowpack check <bundle>
knowpack graph <bundle> --out graph.json

Do not ship yet

  • Google Cloud Knowledge Catalog integration.
  • BigQuery auth and billing flow.
  • Agentic web crawler.
  • Hosted SaaS.
  • Custom graph database.
  • Formal ontology editor.
The lazy useful path: file parser, validator, indexer, graph JSON, HTML renderer. Add MCP only after the bundle format works reliably.

Already Existing Things

Project Similar to OKF? What it gives us Gap we can fill
GoogleCloudPlatform OKF Closest Draft spec, reference agent, tests, sample bundles, visualizer. Standalone polished CLI, validator, package, docs, fixture suite, non-Google examples.
llms.txt Adjacent Markdown convention for giving LLMs curated website context; has parser/integration ecosystem. OKF is deeper: many concept files, frontmatter, links, citations, graph, and bundle validation.
Backstage Software Catalog Adjacent Metadata YAML files in repos, harvested into a developer portal. OKF can be lighter, more prose-heavy, and agent-first without deploying Backstage.
OpenMetadata Adjacent Large open metadata standard with JSON Schemas, RDF/OWL, APIs, governance, quality, lineage. OKF can be simpler and repo-native; OpenMetadata is a full platform standard.
DataHub Adjacent Open-source metadata platform and commercial context platform for AI agents. OKF can be a portable bundle format that does not require running a platform.
OpenLineage Narrow Open standard for lineage metadata events around jobs, runs, and datasets. OKF can reference lineage, but is not limited to runtime events.
CKAN and Data Package Inputs Open data portals and dataset/resource/schema packaging standards. OKF can explain how an agent should use data, not just publish or package datasets.
OpenAPI, JSON Schema, RDF, JSON-LD Standards Formal machine-readable schemas, API descriptions, and graph data models. OKF should reference these, not replace them. The gap is agent-readable context around them.

Design For Our Tool

Repository shape

okf-tool/
  packages/
    okf-core/        # parser, validator, graph
    okf-cli/         # command line
    okf-render/      # static HTML renderer
    okf-mcp/         # later
  fixtures/
    valid-basic/
    broken-links/
    missing-type/
    citations/
  docs/
    getting-started.md
    spec-notes.md

Core validation rules

  • Every concept is a non-reserved .md file.
  • Every concept has parseable YAML frontmatter.
  • Every concept has non-empty type.
  • index.md and log.md are reserved.
  • Unknown keys are preserved and allowed.
  • Broken links warn, not fail by default.
  • External citations warn when missing from claim-heavy files.
Implementation choice: TypeScript is likely best if we want one package for CLI, web renderer, and MCP. Python is better if the first converter target is data catalogs and notebooks. Pick one, do not build both.

Why Models Need This

Context

Weights are stale

The model does not know your current tables, APIs, policies, metrics, product rules, or changed meanings unless those are retrieved.

Meaning

Raw metadata is thin

A schema says revenue NUMERIC. OKF can say what revenue means, which date to filter, and which table is deprecated.

Trust

Citations matter

Agents need source links, provenance, caveats, and update dates to separate supported answers from guesses.

Next Seven Actions

  1. Create a tiny sample OKF bundle for a non-database project: API, policy, playbook, metric, reference.
  2. Write a validator spec as tests before writing the CLI.
  3. Implement validate, index, and graph.
  4. Render a static HTML viewer from graph JSON and Markdown content.
  5. Add one converter: OpenAPI to OKF or CKAN/Data Package to OKF.
  6. Add an MCP read-only server after local validation works.
  7. Publish as a small open-source project only when it can pack context and pass its own golden retrieval tests.
